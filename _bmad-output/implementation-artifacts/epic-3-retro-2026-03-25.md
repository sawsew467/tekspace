## 📚 Lessons Learned

### Technical

1. **UX redesign giữa code review không làm chậm epic nếu được handle tốt.** Story 3.1 pivot sang heatmap trong code review — kết quả tốt hơn nhưng để lại dead code. Pattern: khi redesign mid-story, explicitly mark old code for removal.

2. **Client-side "who is online" tốt hơn server-side endpoint.** `getOnlineMemberIds()` là pure UTC comparison từ data đã fetch — không cần API call riêng. Đơn giản, testable, và đủ accurate cho use case này.

3. **`>= 4 members` threshold cho anonymous comparison là good default.** Dưới 4 người thì individual data quá dễ infer. Pattern này nên reuse cho bất kỳ aggregate comparison nào trong tương lai.

4. **Recharts đã trong bundle từ Story 5-2** — không phải thư viện mới cần lo ngại cho Story 5-3.

### Process

5. **"Intelligence từ story trước" là pattern có giá trị cao nhất trong Epic 3.** Nên trở thành standard section trong mọi story spec từ đây trở đi.

6. **Story file header là problem của quy trình, không phải của agent.** Ba lần miss chứng minh rằng reminder trong retro không đủ — cần embed rule vào story spec template hoặc SM checklist bắt buộc.

7. **Parallel epic model cần explicit lesson propagation mechanism.** Retro insights chỉ có giá trị nếu được embed vào active story specs, không phải chỉ vào retrospective document.
