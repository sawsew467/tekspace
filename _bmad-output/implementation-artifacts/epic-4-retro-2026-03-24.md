## 📚 Lessons Learned

### Technical

1. **DB-first foundation pays off** — `daily_reports` table được setup đầy đủ từ trước giúp Epic 4 focus 100% vào FE. Đây là pattern đáng duy trì: chuẩn bị DB schema cho epic N+1 trong epic N nếu biết trước.

2. **`useWatch` thay vì `form.watch()` trong React Hook Form** — bắt buộc khi dùng React Compiler. `form.watch()` gây "incompatible-library" warning. Pattern: `useWatch({ control, name })` + sub-component nếu trong loop.

3. **Client-side merge > Complex SQL JOIN** — Supabase PostgREST không support LEFT JOIN từ một table sang table khác cho "missing rows". Client-side merge với Map lookup là pragmatic và maintainable.

4. **Service file editing risk** — Khi edit file có nhiều functions, verify function list trước và sau khi edit để tránh silent deletion.

5. **`hasDiscrepancy` pure function pattern** — Business logic phức tạp nên được extract thành pure function riêng, testable độc lập. Pattern này có thể reuse cho future validation logic.

### Process & Design

6. **"Spec compliance" ≠ "Good UX"** — AI dev agent deliver đúng acceptance criteria nhưng không tự detect UX issues. Cần một UX review pass (có thể là Thắng review screenshot) sau mỗi epic, không chỉ ở retro.

7. **Product decisions cần được captured trong retro** — Retro này phát sinh 3 product decisions quan trọng (per-task hours, edit window, unlimited history). Những decisions này cần được documented và trở thành stories cho Epic 5.

8. **Story close convention vẫn chưa nhất quán** — Hai epic liên tiếp (1 và 4) đều có story files không được update status. Cần automation hoặc checklist cứng hơn.
